Observation Identification Service API specifications

Notes on terminology

  • Identification model implementations are published instead of models (a model implementation is a combination of one or more models and a configuration of how to use them)

API endpoints

The v1 API is the first stable version of the API. Updates to the v1 API will be backwards compatible with existing v1 applications.

  • <tag> refers to a manually assigned name for a specific model implementation


Identify

/v1/observation/identify

Common endpoint for all model implementations


/v1/observation/identify/<tag>

Endpoint for specific implementation

Taxa

/v1/observation/taxa/<tag>

End point for taxa in implementation


/v1/observation/taxa/<tag>?id=<id>

End point for taxa in implementation filtered by taxon id

Documentation

/v1/observation/documentation/<tag>

End point for documentation of implementation (description, release notes, etc.)

  • The response field identification_advice has been removed

    • this responsibility is left to the client

  • The response field success has been removed

    • replaced by HTTP (error) codes and specific error messages

  • The predictions field vernacular_name has been removed

  • The response field version has been moved into the response object model_implementation

New: preliminary location support

Introduction


The location of an observation is a valuable piece of information for automated species identification. For example species identification can likely be improved by including probabilities from distribution models in the recognition. The location is also important to warn the user that a certain identification model is probably not suitable for the location of the observation. As our current observation models are based on Dutch/Belgium data, they currently do not apply to geographical areas located far away from the Netherlands and Belgium. The goal of this section is to adapt functional design of the API to support location information mainly for future use.

Implementation for API v1

For computation of the response field identification.location.confidence at first release of v1 we use the following method:

  • The (estimated) geographical center of the image recognition model is determined

    • fixed to the GPS coordinates of Boxtel (51.59, 5.33) for the Dutch/Belgian model

  • The estimated geographical radius in GPS degrees of the image recognition model is determined

    • fixed to 2 degrees for the Dutch/Belgian model


The numerical value of identification.location.confidence is then

  • 1.00 for location_coordinates within a distance of 0 to radius degrees of the center

    • ca. 0-226 km for Dutch/Belgian model

  • 0.75 for location_coordinates within a distance of radius to (radius + 1) degrees of the center

    • ca. 226-339 km for Dutch/Belgian model

  • 0.50 for location_coordinates within a distance of (radius + 1) to (radius + 2) degrees of the center

    • ca. 339-452 km for Dutch/Belgian model

  • 0.25 for location_coordinates within a distance of (radius + 2) to (radius + 3) degrees of the center

    • ca. 452-565 km for Dutch/Belgian model

  • 0.00 for location_coordinates within a distance of > (radius + 3) degrees of the center

    • ca. > 565 km for Dutch/Belgian model

  • -1.00 if location_coordinates is not given or computation fails


The API documentation will state


“identification.location.confidence indicates the confidence of the model for the supplied geographical location as a number between 1.0 (high confidence) and 0.0 (no confidence). The location confidence and image confidence are independently calculated in this version of the API”.


Note on future versions


The exact implementation of the numerical value is likely to change in updates of the API. The syntactic definition of the field identification.location.confidence will not change.


In later versions the per class (taxon) probabilities will be updated based on location. This feature is independent from what is written here.


Implementation notes

  • Extraction and validation of GPS coordinates from photos is done by the client. The demonstrator will not support this feature in v1

  • No server side validation, parsing or clean-up of location_coordinates is done in v1

  • The proposed method is likely not useful for much else than the Dutch/Belgian model. As soon as we have other models we can redefine the calculation

  • Interpretation and acting upon the identification.location.confidence value is for the implementing application
